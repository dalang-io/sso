What is Dalang SSO

Dalang SSO is a self-hosted, open-source OAuth 2.0 / OpenID Connect identity provider written in Rust — a lightweight alternative to Keycloak. You register OAuth clients in a dashboard, get a client_id and client_secret, and your apps sign users in against your own domain. It runs as a single static binary with an embedded SQLite database, works airgapped, and ships client SDKs for six languages.

Quickstart

1 · Run the server

# Pre-built binary (see the GitHub Releases page for your platform)
curl -L https://github.com/dalang-io/sso/releases/latest/download/sso-linux-x86_64 -o sso
chmod +x sso && ./sso

# …or build from source (Rust 1.90+)
git clone https://github.com/dalang-io/sso && cd sso
cargo run -p sso-server

Open http://localhost:8080. The first visit sends you to /setup to create the super admin (one-time onboarding).

2 · Create an OAuth client

In the dashboard: create a client, add an Authorized redirect URI (e.g. https://app.example.com/callback), and copy the client_id + client_secret (shown once).

3 · Sign users in

Use an SDK (below) or hit the endpoints directly. The default flow is Authorization Code + PKCE.

Tenants & roles

Dalang SSO is multi-tenant. A tenant is an isolated workspace that owns its OAuth clients. Dashboard users (members) have a role:

RoleScopeCan do
superGlobalManage tenants & members; all clients/secrets everywhere.
managerOwn tenantCreate/delete clients, edit config, manage secrets.
developerOwn tenantAdd/delete client secrets only (no client CRUD).

Every client can hold up to two secrets for zero-downtime rotation, and can be restricted to specific end-user emails (e.g. @yourco.com).

The OAuth flow

App ──1 authorize URL──▸ Browser ──▸ /oauth/authorize   (user logs in + consents)
App ◀─2 redirect ?code=… ── Browser
App ──3 POST /oauth/token (code + client_secret + PKCE) ──▸ SSO
App ◀─4 access_token + id_token + refresh_token (JWT) ────── SSO
App ──5 GET /oauth/userinfo (Bearer access_token) ────────▸ SSO ──▸ { sub, email }

Access and id tokens are self-verifying RS256 JWTs — verify them against the JWKS without a round-trip. Refresh tokens rotate on every use.

SDKs

Official clients live in sdks/ — all expose the same shape: build an authorize URL (with PKCE), exchange the code, refresh, and fetch UserInfo.

JavaScript / TypeScript

import { DalangSSO, generatePkce } from "@dalang-io/sso";

const sso = new DalangSSO({ baseUrl: "https://sso.example.com",
  clientId: "CLIENT_ID", clientSecret: process.env.SSO_SECRET,
  redirectUri: "https://app.example.com/callback" });

// 1. redirect the user
const pkce = await generatePkce();
location.href = sso.authorizeUrl({ scope: "openid email", state, pkce });

// 2. on your callback (server-side)
const tokens = await sso.exchangeCode(code, pkce.verifier);
const me = await sso.userInfo(tokens.access_token);

Python

from dalang_sso import Client, generate_pkce
sso = Client("https://sso.example.com", "CLIENT_ID", SECRET, "https://app/callback")
pkce = generate_pkce()
url = sso.authorize_url("openid email", "state", pkce)
tokens = sso.exchange_code(code, pkce)
me = sso.userinfo(tokens.access_token)

Go

sso := dalangsso.New("https://sso.example.com", "CLIENT_ID", secret, "https://app/callback")
pkce, _ := dalangsso.GeneratePkce()
url := sso.AuthorizeURL("openid email", "state", pkce)
tokens, _ := sso.ExchangeCode(code, pkce)
me, _ := sso.UserInfo(tokens.AccessToken)

Rust, Java and PHP follow the same API — see each SDK’s README.

Self-hosting

Dalang SSO is one static binary — copy it to a server, set a .env, and run it under systemd behind a TLS-terminating reverse proxy (or Cloudflare). Point SSO_PUBLIC_URL at your HTTPS origin; it becomes the OIDC issuer. Scale horizontally: the app tier is stateless (JWTs verify without the DB; the admin session is a signed cookie), so run N nodes behind a load balancer with a shared Postgres/MySQL and the same signing key.

Post-quantum

Set SSO_TOKEN_SIGNING_ALG=ml-dsa-65 to sign access/id tokens with ML-DSA (FIPS 204), making them resistant to quantum attacks. The JWKS advertises the key and discovery reports the algorithm. The classical rs256 default maximizes interop with existing JWT libraries. Password/secret hashing (Argon2) and token hashing (SHA-256) are already quantum-safe.

FAQ

Is this a drop-in Keycloak replacement?

It covers the core IdP surface (clients, login, consent, tokens, JWKS, UserInfo, RBAC, multi-tenancy). It is intentionally lean — it is not trying to match every Keycloak feature (e.g. SAML, LDAP federation), but it is far smaller and faster to operate.

Does it run without internet access?

Yes — no CDN or external fonts; all assets are local, and SQLite is embedded.

How do I build from source?

cargo build --release -p sso-server (Rust 1.90+). Cross-compile a static Linux binary with cargo zigbuild --release --target x86_64-unknown-linux-musl -p sso-server.
